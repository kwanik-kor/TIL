# 1부 AI 프로젝트를 시작하며: 기본 점검 

## 01. 프로젝트가 실패할 확률을 추산해보았나?

2024년 S&P 글로벌 마켓 인텔리전스(Global Market Intelligence)
- AI 프로젝트의 42% 완전 폐기
- 가트너(Gartner) 27년까지 에이전틱 AI 프로젝트의 40% 이상이 취소될 것이라는 전망

**생성형 AI/LLM을 이용한 프로젝트가 인기가 많은 이유**
- 자연어를 이용하기 떄문
- 대규모 학습이 필요 없는 제로샷(zero-shot) 또는 퓨샷(few-shot) 프롬프팅으로 무언가를 할 수 있음
- 모델이 범용으로 쓰일 수 있음
- 멀티 모달(multi modal)이 됨
	- 입출력 강화에 따라, 텍스트 출력 뿐 아니라 이미지 생성 등 다양한 형태의 데이터에 접근이 가능해짐
- 복잡한 일을 덜어낼 수 있음

**생성형 AI/LLM 프로젝트들의 치명적 함정들**
- 데이터가 편향될 수 있음
- 평가 지표를 잘못 선택할 수 있음
	- BLEU(Bilingual Evaluation Understudy): 생성 텍스트 품질 평가지표, 생성물이 참조문과 얼마나 많은 n그램을 정확히 일치시키는가
	- ROUGE(Recall-Oriented Understudy for Gisting Evaluation): 텍스트 요약이나 번역 품질 평가 지표, 생성된 텍스트가 사람이 만든 정답과 얼마나 많은 단어나 구절을 공유하는지 측정
	- + 실제 일을 주고 일의 결과를 가지고 점수를 내는 방식을 사용해야 함
- 환각 현상
- 데이터의 법적 문제

## 02. 생성 AI 기술은 절대반지일까?

> 뉴질랜드 팩엔세이브(Pak'nSave)의 잘못된 레시피 추천
> 뉴욕시 비즈니스 상담 챗봇의 부적절한 상담내용

**기존 머신러닝/규칙기반 기술이 더 적합한 문제도 있다**
- 정밀한 수치 계산이 필요한 경우
	- 패턴 기반 추론을 사용함에 따라 복잡한 수치 연산에서는 상대적으로 더 높은 오차율을 보임
	- 절대적인 정확성이 필요한 경우 팩트체크 서비스를 이용해 가드레일을 구축하거나 사람 리뷰를 거치게 하는 편이 안전함
		- 구글 팩트체크(Google FactCheck)
		- 이머전트(Emergent): 실시간 미확인 정보, 소문 추적/검증 플랫폼
- 의사결정 지원 시스템(DSS, Decision Support System)
	- 병원에서 약물 부작용 및 중복 처방에 대한 경고를 보내는데, 이 경우 규칙기반 시스템이 더 나을 수 있음
	- CDSS(Clinical DSS)
- 추천 시스템
	- 기존 추천 알고리즘이 더 나을 수 있음. 추천에 적합하게 RAG를 튜닝하기 복잡함
	- AWS Personalize

**생성 AI/LLM이 더 적합한 문제**
- **자연어 처리 작업**
	- 텍스트 분석, 추출, 분류와 같은 작업은 기존에 복잡한 머신러닝이나 딥러닝 파이프라인이 필요했음
- 복잡한 과학적 분석 업무나 문헌 자료를 정리하고 요약하는 작업
- 검증이 쉬운 문제를 해결하는 경우
	- 객관적 판단 기준이 있는 문제
		- 코드의 실행/실패, 수학 문제의 오답여부
	-  즉각적인 피드백
		- 결과를 바로 확인할 수 있어, 시행착오를 통한 빠른 개선이 가능
	- 역할 분담
		- AI가 사람이 기억하기 어려운 세부사항을 처리해주고, 사람은 전체적인 방향성 설정과 검증에 집중

## 03. 업무 노하우와 도메인 지식을 파악해 적용했는가?

**'라스트 마일 문제'**
- AI는 특정 업무에서 정말 탁월하지만 전체 프로세스를 처음부터 끝까지 완벽하게 처리하지는 못한다.

> 회사가 가진 소프트웨어가 고객에게 어떤 가치를 주는지, 시장에서 어떤 문제를 해결하는지와 같은 본질적인 경쟁력만 남게 된다.
> 결국 남는 것은 **도메인 지식과 숨은 노하우**

**도메인 지식과 숨은 업무 노하우를 끌어내는 방법**

### **중요 결정 방법론(CDM, Critical Decision Method)**
- 전문가의 의사결정 과정을 체계적으로 분석하는 기법
- 사건이나 상황을 심층적으로 조사하여, 전문가와 초보자 사이의 차이점을 명확히 이해하며, 이를 바탕으로 개선 방안과 도메인 지식과 숨겨진 업무 노하우를 도출하는 것
- 숨겨진 사고 과정을 체계적으로 끄집어 내야 함

1) **분석할 중요한 사건 선정하기**
- 특별한 조건을 만족하는 사건을 골라야 함
	- 비일상적인 사건
	- 도전적인 사건: 어려움이 있었던 사건
	- 내공이나 숨겨진 전문성이 드러나는 사건

1) **시간 순서표 만들기**
- 해당 사건에서 일어난 일들을 시작부터 끝까지 시간 순서대로 정리해야 함
- 모든 일을 기록하는 것이 아니라 중요한 분기점들을 기록해야 함
	- '처음에는 이런 상황인줄 알았는데, 나중에 알고 보니 완전히 다른 상황이었던 적이 있나요?'

1) 깊이 파고들기
- '당연한 일'이라고 답하더라도, 오랜 경험ㅇ르 통해 체득한 특별한 판단 능력이나 노하우가 있을 것이고 이것을 끌어낼 수 있어야 함

1) 가정해보는(What If) 질문하기
- 초심자는 매뉴얼대로 일하거나 겉으로 보이는 정보만으로 판단하는 경우가 많음
	- 만약 초보자가 이 상황을 마주했다면 어떤 오류를 범했을까?
	- 왜?
	- 만약 상황의 일부 조건이 달랐따면, 결정/판단/행동/계획에 어떤 영향을 주었을까?

| 의사결정사항 | 무엇이 이 결정을 어렵게 만들었는가? | 종종 어떤 종류의 오류가 만들어졌는가? | 초보자와 달리 전문가는 어떻게 이런 결정을 할 것인가? |
| ------ | -------------------- | --------------------- | ------------------------------ |
|        |                      |                       |                                |

### 암묵지를 AI가 이해하고 자동화할 수 있도록 만들자
> 전문가들만 알고 있는 절차들 속에는, 워크플로나 에이전트를 설계하는 데 바로 사용하는데 필요한 데이터들이 있을 것이고, 이것을 골든 테이블 설계나 데이터를 가져오라는 프롬프트 설계에 이용할 수 있음


- 우리가 풀어야 할 가치 있는 문제가 무엇인가?

- 같은 암묵지를 가지고 있는 사람이나 회사가 있다면 생성 AI/LLM을 통해서 해당 지식이나 노하우를 자동화/최적화 할 수 있는가?


# 2부 AI 프로젝트의 실제 구축 방안
## 04. 우리 비즈니스에 맞는 AI 기술을 제대로 알고 선택했는가?

> RAG(Retrieval-Augmented Generation)
> 긴 문서를 벡터 DB나 검색 엔진에 쪼개 저장해두고, 질문이 들어오면 적절한 부분을 검색해 답변의 근거로 삼는 방식(질문에 맞는 데이터를 가져와서 이것으로 답)

LLM은 사용자가 질문하면 답을 제공하는, 기본적으로 단일한 대화형 시스템이다. 별도의 보조 도구가 없음에 따라 학습 시점 이전의 데이터와 지식에만 의존함에 따라, 이 한계를 보완하기 위해 외부 데이터를 참조하거나 과거 대화를 기억해 답변에 반영할 수 있는 '보조 장치'를 필요로 하게 된다.

결국 제대로 된 답변 혹은 문제 해결을 위해서는 적절한 데이터를 필요로 하는데, 이를 위해 1) 데이터를 가져와서 2) 이것으로 답을 하는 시스템이 필요하고, 이것을 RAG라고 한다.

하지만 작업에 필요한 모든 자료를 일일이 찾고, 검토해서 해결하는 과정은 어려운데, 원하는 답을 얻기 위해 끊임없이 질문을 던져야 하기 때문이다. 해결해야 하는 작업에 대해 누군가 스스로 필요한 질문을 던지고 최종적인 답까지 구해오는 것. 이 과제를 해결하기 위해 등장한 것이 Agent다. **특정한 상황이 주어지면 그 다음의 액션을 스스로 결정하고 실행하는 것**이 에이전트의 특징.

에이전트는 스스로 결정하고 실행하기 위해 '추론(Reasoning)'과 '계획(Plan)'의 과정을 거치는데, 다음의 구조를 띈다.
- 생각의 사슬(CoT, Chain of Thought): 추론 과정을 단계별로 텍스트로 풀어내도록 유도
- ReAct 패턴: 추론(Reasoning) + 행동(Acting)을 번갈아 수행
- 작업 재구성(Task decomposition): 목표를 하위 태스크로 나누는 계획 전략
- 그래프 기반의 계획(Graph-based Planning): 단계들을 그래프 형태로 구성해 유연하게 탐색

에이전트의 문제 해결 가능성에 따라 다양한 제품군과 프레임워크가 등장했으나, 외부 도구에 연결하는 일이 매우 번거로운데, 공식 지원을 기다리거나 직접 통합해야만 했다. 이 통합을 더 쉽게 만들어주는 것이 다음 과제였고, 그래서 **MCP(Model Context Protocol)** 이 등장했다. 앤트로픽에서 제안한 방식으로, 여러 에이전트 라이브러리에서 각 도구들이 통합되기를 기다리지 말고 하나의 표준화된 '프로토콜'에 맞추자는 것.

## 05. 서비스 가능한 AI 프로덕트를 위한 아키텍처 요소를 잘 구성했는가?

프로덕션에서 쓰이는 제품을 만들기 위해 다음의 조건하에서 개발하고 검증
- 사용자의 예상 외의 입력
- API는 0.1% 정도는 실패
- 사용자는 3초 이내에 응답이 오기를 원함
- LLM 이용을 위한 비용은 반드시 예산 범위 내여야 함

### 1) 가드레일
입/출력에 대한 보호장치로, 사용자의 입력부터 LLM의 최종 출력까지 전 단계에 걸쳐 적용되는 다층 방어 전략을 채택해야 함

- **전처리 가드레일(pre-processing guardrails)**
	- 사용자 프롬프트가 LLM에 도달하기 전에 악의적이거나 도메인 경계를 벗어난 입력을 탐지하고 이를 방어하는 단계
	- 보통 프롬프트로 구현
- **생성 중 보호 가드레일(generation-time guardrails)**
	- LLM의 시스템 프롬프트를 보호하고 에이전트의 역할 격리(role isolation)를 유지하는데 중점을 둠
	- 핵심 지침을 무시하지 않도록 상기시키고, 외부 도구 사용 시 최소 권한 원칙(least privilege)을 엄격히 적용해서 도구 오용을 방지해야 함
- **후처리 가드레일(post-processing guardrails)**
	- LLM이 생성한 응답이 최종 사용자에게 도달하기 전에 안정성 및 규제 준수를 검증하고 수정하는 단계
	- 개인 식별 정보 마스킹(PII, personality identifiable information)
		- **탐지(detection)**: NER(named entity recognition)이나 정규 표현식을 사용하여 출력 텍스트 내의 민감 정보 식별
		- **마스킹(masking)**: 식별된 민감 정보를 처리용 플레이스홀더 토큰으로 대체
		- **복원(restoration)**: 시스템 내부적으로 원본 데이터를 유지하고, 외부 API 호출 시에만 마스킹된 데이터 사용
- 지속적인 모니터링
	- 차단된 입/출력 시도
	- 우회 시도(jailbreak attempt)에 대한 상세한 로그와 지표를 지속적으로 모니터링
		- NVIDIA Guardrails - https://github.com/NVIDIA-NeMo/Guardrails
		- Protect AI - https://github.com/protectai/llm-guard
		- Proventra-core - https://github.com/proventra/proventra-core

### 2) 라우터와 게이트웨이

항시 최고 성능의 생성 AI/LLM 모델을 사용할 필요는 없다. 의도 분류, 도구나 데이터를 가져올 소스를 선택하는 것과 같은 경량 프로세스에는 작은 크기의 간단한 모델을 써서 처리해도 충분함.

**게이트웨이(Gateway)**
- 여러 모델들을 하나의 인터페이스로 부를 수 있게 함
- 오픈 AI SDK 인터페이스를 동일하게 쓰게 하고, 이 뒤에 다양한 모델들을 호출할 수 있게 구성(라우터)
- 구성 이유
	- 사용하는 LLM API에 장애가 있을 수 있음
	- 다른 모델을 선택하더라도 게이트웨이 인터페이스를 그대로 따를 수 있다면 다른 부분을 수정하지 않아도 됨
	- 주어진 쿼리를 처리하기 위해서 모델들이 얼마나 토큰을 쓰고 있는지 사용량을 제어하거나 비용제한을 둘 수 있음

### 3) 프롬프트 템플릿 관리

> 프롬프트 템플릿
> 특정 내용이 비어있어서, 쿼리나 컨텍스트들을 주입하면 되도록 미리 만들어 놓은 프롬프트 양식

템플릿에 예제 선택자를 통해, DB에 기록된 예제들 중 현재 질의와 유사도가 가장 높은 예제를 찾아서 해당 예제 정보를 프롬프트 앞에 넣어주게 됨.
- https://paraprompt.xyz
- https://github.com/llmkit-ai/llmkit

### 4) 캐시로 응답시간 줄이기

트래픽이 몰리면 컴퓨터 자원을 자연스럽게 더 많이 사용하게 되는데, 생성 AI/LLM은 웹 서비스에 비해 더 많은 양의 자원을 이용한다. 이에, 자원을 효율적으로 사용하기 위해 캐시레이어 도입을 고민할 수 있다.

- 완전 일치 캐싱(exact caching)
	- 같은 쿼리가 들어올 경우 같은 답변
	- 생각의 사슬, 검색, SQL 실행, 웹 검색처럼 오래 걸리는 동작에서 효과적
	- 개인정보 유출과 같은 데이터 민감성에 주의할 필요
- 시맨틱 캐싱(Semantic caching)
	- 들어온 쿼리들이 의미상 같은지를 판단하여 비슷한 쿼리면 같은 답을 던지는 방식
	- 캐싱을 담당하는 백터 DB가 필요함
	- 백터 DB 검색을 통해 비슷한 것이 있는지 검색 후, 임계치를 넘는 유사도를 보여주면 벡터 DB에 보관중인 답변을 되돌려줌

### 5) 출력을 입력으로 넣어주기

에이전트처럼 출력된 결과를 새로운 컨텍스트로 입력되는 구조
대화 세션 단위인 단기 메모리와 여러 대화 세션들의 기록들을 데이터베이스로 정리해서 기억하고 있는 장기 메모리로 나누어 저장할 수 있음

### 6) 모니터링과 관찰 가능성 추가하기

생성 AI/LLM도 어떤 웹서비스가 됨. 즉, 웹서비스에 필요로 하는 모든 것이 기본적으로 들어가게 되고 여기에 관측가능성(observability)을 더 고민하게 됨.
- 서비스가 잘 돌아가는지에 대한 관측
- 모델로 답변을 제대로 주고 받고 있는가
- 답변 성능은 제대로 나오는가
- 모델이 쓰이는 비용은 얼마인가
- 프롬프트는 어덯게 관리되고 있는가
- 서비스 수준 계약(SLA, Service Level Agreement)을 보장할 것인가
- 평가는 어떻게 할 것인가
- LLMOps
	- Langfuse(https://langfuse.com)
	- 랭퓨즈를 통합 할 시, 이 안에서 생성 AI/LLM 앱이 어떻게 동작하고 있는지를 단계별로 시각화해서 관찰할 수 있음
	- 모델을 이용하는 비용과 지연상황 등도 모니터링할 수 있음
	- 프롬프트에 대한 버전 컨트롤이 가능하므로 프롬프트 관리가 쉬워짐
	- 평가 기능이 내장되어 있으므로 CI/CD에 통합해서 생성 AI/LLM 앱의 성능 평가에 도움 받을 수 있음
	- 사람이 생성 AI/LLM 앱의 응답을 평가할 수 있는 기능을 이용할 수 있음
